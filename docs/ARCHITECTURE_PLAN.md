# Architecture Plan: VLM + Navigation + Action Expert for LIMO Cobot

## 全体アーキテクチャ

本研究では、自然言語命令から未知環境の探索、状況理解、行動選択、アーム操作までを階層的に分担する。全体像は次のように整理できる。

```text
Natural Language Command
例: 「崩落現場で人命救助を行って」

↓
LLM Planner / Task Planner
- 高レベル計画
- 探索方針
- 次に何を見るか
- 何をどかすか
- いつ停止するか

↓
Scene Memory / Map
- 探索済み領域
- 障害物
- 瓦礫
- 被災者候補
- 危険領域

↓
VLM Perception
- 瓦礫認識
- 人の一部らしき領域の認識
- 掴めそうな物体の判断
- 触ってはいけない領域の判断

↓
Action Selector
- navigate
- inspect
- grasp
- push
- remove
- stop / ask human

↓
Navigation Model
例: Uni-NaVid / Isaac Sim
- 未知環境探索
- 目標地点への移動
- 障害物回避

+
Action Expert
例: ACT / VLA / Diffusion Policy
- アームを伸ばす
- 掴む
- 押す
- 引く
- どかす

↓
LIMO Pro2 + myCobot280
```

## 重要な考え方

本研究では、1つの巨大モデルをゼロから作るのではなく、既存モデルと自作action expertを統合するアーキテクチャを作る。

研究全体としては「モデルを作る」よりも「アーキテクチャを作る」に近い。既存のVLM、LLM、ナビゲーションモデルを活用しつつ、LIMO Pro2 + myCobot280という実機構成で動く低レベル操作部分を自作データで補う。

- myCobot280用のACT/action expertは自作データで学習する
- 車体側はUni-NaVidなどを用いて移動・探索を行う
- アーム側はACT/action expertで把持・除去動作を行う
- VLM/LLMは「何を操作するか」「危険かどうか」「次に何を試すか」を判断する
- Action Selectorは、移動、観察、把持、押し操作、除去、停止確認を切り替える

## 車体とアームの統合で難しい点

### 難しい点1: LIMO Pro2とmyCobot280が別々の身体要素である

LIMOは移動、myCobotは操作を担当するが、最終的には1つのロボット身体として扱う必要がある。

車体の位置、アームの可動範囲、カメラ視点、作業対象の位置関係を統合しなければ、移動後にアームが届かない、視点がずれる、対象物が画角外になるといった問題が起きる。

### 難しい点2: 車体側モデルがアームの存在を考慮できるか

通常のナビゲーションモデルは障害物を避ける対象として扱う。しかし、アーム付き移動ロボットでは障害物を「どかせる対象」として扱う必要がある。

```text
通常の移動ロボット:
障害物がある
↓
迂回する

アーム付き移動ロボット:
障害物がある
↓
迂回する or アームでどかす
```

この違いにより、単なる経路計画ではなく、環境を操作してから移動するという選択肢が必要になる。

### 難しい点3: どのタイミングでアームを使うか

アームを使うかどうかは、障害物の種類、重さ、位置、人や危険領域との距離によって変わる。

- 迂回できるなら迂回する
- 迂回できないが軽い物体ならどかす
- 人の一部に近い場合は慎重に扱う
- 危険なら止まって人間に確認する

この判断は、VLM/LLM、Scene Memory、ルールベースの安全条件、action expertの成功可能性を組み合わせて行う必要がある。

## ファインチューニングの考え方

車体が動くため、アーム側は移動後の位置ずれや視点変化に対応できるようにする必要がある。机上で固定した把持だけでなく、車体の停止位置が少しずれた状態でも対象物へ接近できるデータが必要になる。

また、アームが搭載されているため、車体側は「どかせる障害物」を考慮した移動方策に拡張する必要がある。将来的には、車体側とアーム側を別々にファインチューニングするのではなく、行動選択レベルで統合する必要がある。

初期段階では、アーム操作をACT/action expertとして独立に学習し、車体移動は既存ナビゲーションモデルや手動配置で補う。その後、Action Selectorを通して、移動、観察、把持、除去を接続する。
